1. What Is Y-Lang?

Y-Lang is a systems programming language designed to provide programmers with direct access to GPU hardware. It avoids abstractions and compiler heuristics, offering a typed vocabulary that maps directly to the hardware's capabilities.



Most high-performance languages trade off expressiveness for beginners or portability across GPU generations. Y-Lang makes neither concession. It is precise, hardware-specific, and intended for programmers familiar with concepts such as MMA atoms who require direct hardware control.




1.1 The Problem Y-Lang Solves

Modern GPU kernel languages occupy two camps:





High-level (Triton, Mojo): The compiler makes decisions about tile sizes, memory layouts, pipeline depth, and instruction selection. When it guesses right, performance is good. When it guesses wrong, you have no way to correct it without abandoning the language entirely.



Low-level (CUDA C + PTX): You have full control but no safety. A wrong register fragment role, a missing barrier after cp. async, a swizzle function the compiler silently re-optimizes — these are silent runtime bugs that only appear in a profiler hours later.



Y-Lang occupies a third position: full low-level control with a type system that catches silent bugs at compile time. The programmer expresses silicon intent; the compiler verifies and faithfully emits it.




1.2 The Core Philosophy





Director-Level Synthesis

You are not writing a matrix multiply. You are directing a sequence of warp-collective fragment distributions across typed memory tiers, backed by compiler-verified synchronization proofs. The language is the documentation.





2. Why Y-Lang May Outperform Existing Languages

This section compares Y-Lang against CUDA C, Triton, and Mojo across the four dimensions that determine real-world kernel performance.




2.1 Shared Memory Bank Conflicts — Swizzling as a Type

On an RTX 4070, shared memory has 32 banks, each 4 bytes wide. When multiple threads in a warp are on an RTX 4070, shared memory consists of 32 banks, each 4 bytes wide. When multiple threads in a warp access the same bank simultaneously, accesses are serialized, resulting in a bank conflict that reduces shared memory throughput. Lice access land on different banks. Every high-performance GEMM kernel does this. The differences lie in how each language handles it:





CUDA C: You write the swizzle as a function. The compiler sees opaque integer math and may reorder surrounding memory operations in ways that break your swizzle's assumptions. No compile-time verification.



Triton: The compiler's tiling engine handles layout internally. You cannot express a custom swizzle pattern at all in standard Triton.



Mojo: MLIR canonicalization passes can strength-reduce integer arithmetic. Your swizzle XOR may be transformed in ways that change the bit pattern. Y-Lang: Swizzle is a type parameter. The compiler recognizes the swizzled layout, maintains it through all transformations, and statically verifies conflict-freedom for all warp-collective operations on the tile.



// Y-Lang: swizzle is part of the memory TYPE, not a function

type ATile = SmemLayout&lt;

    dtype:   F16,

    rows:    16,

    cols:    64,

    swizzle: Swizzle&lt;XOR_bits=3, base_shift=3, offset=3&gt;

&gt;



// Compiler proves at instantiation:

// forall threads i,j in warp where i != j:

//     bank(ldmatrix_addr(i, ATile)) != bank(ldmatrix_addr(j, ATile))

// Conflict detected? Compile error. Not a profiler finding.




2.2 L2 Cache — Managed, Not Transparent

The RTX 4070 has 36 MB of L2 cache running at roughly 5 TB/s — ten times the bandwidth of VRAM. In AI workloads, weight matrices are reused across many activation tiles. If weights are evicted from L2 between uses, performance falls off a cliff.



Triton and Mojo treat L2 as a transparent hardware detail. Y-Lang treats it as a managed memory tier with explicit lifecycle annotations:



// Tell the compiler this weight tile will be reused N times

// Emits: ld.global.L2::evict_last (persists in L2)

@cache_policy(L2_PERSIST, reuse_count=8)

let weight_tile: L2Memory&lt;F16&gt; = load(weight_ptr);



// One-time activation — evict immediately after use

// Emits: ld.global.L2::evict_first

@cache_policy(L2_EVICT_FIRST)

let act_tile: L2Memory&lt;F16&gt; = load(activation_ptr);



This prevents cache thrashing—the scenario where activation loads displace weight tiles that will be needed again in 4 microseconds.




2.3 Async Pipeline — Scheduled, Not Hoped

Double-buffering is the single most impactful optimization in high-performance GEMM: while the tensor cores compute on tile N, cp. async is already loading tile N+1 into the other SMEM buffer. When the pipeline is correctly sequenced, the tensor cores never idle waiting for memory.



In Triton, the num_stages parameter activates an automated software pipelining pass. When it works, it works well. When the heuristic fails — wrong tile shape, unusual memory access pattern, unexpected register pressure — performance degrades with no diagnostic output and no recourse.



In Y-Lang, pipeline synchronization is a type of obligation. The Transfer type carries a synchronization proof that must be discharged before the destination can be read:



// Transfer&lt;From, To, Policy&gt; is a first-class type

let tx: Transfer&lt;Global, Shared, Async&lt;commit=1&gt;&gt; =

    cp_async(gmem_ptr, smem_tile);



// Compiler enforces: tx must wait before smem_tile is read

// Reading smem_tile before wait(tx) is a TYPE ERROR, not a race

pipe.wait(tx);     // emits: cp.async.wait_group 1

barrier::sync();   // emits: bar.sync



// Only now is smem_tile safe to read

let frag_A = ldmatrix(smem_tile);




2.4 MMA Atoms — Fragment Roles as Types

The m16n8k16 MMA atom is a warp-collective operation. All 32 threads participate, each holding a non-contiguous slice of the A, B, and C matrices known as a fragment. The fragment layout is fixed by the PTX ISA — thread 0 holds specific elements, thread 1 holds different ones, and so on.



In CUDA C, fragments are typed structs, but their roles (A vs. B vs. C) are not enforced by the type system. Passing a B fragment where an A fragment is expected compiles silently and produces incorrect numerical results. In Y-Lang, fragment roles are phantom types:



// Fragment&lt;Op, Role, dtype&gt; — roles are incompatible types

let frag_A: Fragment&lt;MMA_m16n8k16, A, F16&gt;;  // holds r0,r1,r2,r3

let frag_B: Fragment&lt;MMA_m16n8k16, B, F16&gt;;  // holds r4,r5

let frag_C: Fragment&lt;MMA_m16n8k16, C, F16&gt;;  // holds d0,d1



// mma_sync only accepts correct roles — enforced at compile time

let frag_D = mma_sync(frag_A, frag_B, frag_C);



// mma_sync(frag_B, frag_A, frag_C) → COMPILE ERROR

// 'expected Fragment&lt;_, A, _&gt;, got Fragment&lt;_, B, _&gt;'

3. The Y-Lang Type System

Y-Lang's type system has five tiers, each corresponding to a distinct hardware concept. Types from different tiers are incompatible — there are no implicit conversions, only explicit typed operations.




3.1 Memory Space Types

Memory spaces are distinct types with bandwidth and latency metadata embedded at the type level. The compiler uses this metadata to reason about pipeline feasibility before emitting any code.



type GlobalMemory    // HBM2e, ~600 GB/s,  80-120 cycle latency

type L2Memory        // 36MB,  ~5 TB/s,    30-40  cycle latency

type SharedMemory    // 48KB/SM, ~20 TB/s, 1-2    cycle latency

type RegisterFile    // 64KB/SM, ~80 TB/s, 0      cycle latency



// These cannot be cast to each other.

// Cross-tier movement requires an explicit Transfer operation.




3.2 Transfer Types — Synchronization Obligations

Moving data between memory tiers produces a Transfer value. This value is a linear type: it must be consumed exactly once via a wait call. Both forgetting to wait and waiting twice are compile-time errors.



type Transfer&lt;

    from:   MemorySpace,

    to:     MemorySpace,

    policy: Async&lt;commit_group: u32&gt; | Sync,

    width:  u32    // 32 | 64 | 128 bits — controls cp. async instruction width

&gt;



// Transfer is a LINEAR TYPE — must be consumed exactly once

// Dropping tx without wait(tx) → compile error: 'unused transfer obligation'

// Calling wait(tx) twice       → compile error: 'transfer already consumed'




3.3 Layout Types — Swizzle as a Type Property

type SmemLayout&lt;

    dtype:   F16 | BF16 | TF32 | F32,

    rows:    const u32,

    cols:    const u32,

    swizzle: Swizzle&lt;XOR_bits: u3, base_shift: u3, offset: u3&gt;

           | NoSwizzle

&gt;



// Compiler proof obligation at instantiation:

// For every warp-collective op on this layout,

// prove conflict_free(op, layout) = true.

// Failure → compile error with exact conflict report.




3.4 Fragment Types — Warp-Distributed Register Values

type Fragment&lt;

    op:   MMA_m16n8k16 | MMA_m16n8k8 | ...,  // the MMA atom

    role: A | B | C | D,                       // which operand

    dtype: F16 | BF16 | TF32 | F32

&gt;



// Fragment is a PHANTOM TYPE — zero runtime overhead

// Role is enforced at compile time only

// Fragment values cannot be indexed — no frag[i] syntax

// They are only valid as inputs to their matching MMA op




3.5 Pipeline Types — Double-Buffering as a Type

type Pipeline&lt;

    stages:    const u32,          // 2 = double-buffer, 3 = triple

    layout:    SmemLayout&lt;..&gt;,

    invariant: NoOverlap&lt;compute, transfer&gt;  // compiler-verified

&gt;



// NoOverlap invariant: the compiler proves that for every iteration,

// the transfer for stage N+1 does not read memory being written

// by the compute stage for stage N.

// Violation → compile error.





4. Y-Lang Syntax

Y-Lang syntax is designed to be readable without being verbose. The guiding principle: hardware intent must be explicit; boilerplate must not exist.




4.1 Kernel Declaration

// Hardware target is declared at the kernel level

// All type proofs are discharged against this target

@target(RTX_4070)

kernel matmul(

    A: GlobalMemory&lt;F16, shape=[M, K]&gt;,

    B: GlobalMemory&lt;F16, shape=[K, N]&gt;,

    C: GlobalMemory&lt;F32, shape=[M, N]&gt;

) {

    // kernel body

}




4.2 Memory Allocation

// Typed SMEM allocation — swizzle is part of the type

type ATile = SmemLayout&lt;F16, rows=16, cols=64, swizzle=Swizzle&lt;3,3,3&gt;&gt;;

type BTile = SmemLayout&lt;F16, rows=16, cols=32, swizzle=Swizzle&lt;3,3,3&gt;&gt;;



let smem_A: ATile = SharedMemory::alloc&lt;ATile&gt;();

let smem_B: BTile = SharedMemory::alloc&lt;BTile&gt;();



// Accumulator lives in register file — type knows this

let acc: Fragment&lt;MMA_m16n8k16, D, F32&gt; = Fragment::zero();




4.3 The K-Loop — Full Pipeline

@target(RTX_4070)

kernel matmul(A: GlobalMemory&lt;F16&gt;, B: GlobalMemory&lt;F16&gt;, C: GlobalMemory&lt;F32&gt;) {



    type ATile = SmemLayout&lt;F16, rows=16, cols=64, swizzle=Swizzle&lt;3,3,3&gt;&gt;;

    type BTile = SmemLayout&lt;F16, rows=16, cols=32, swizzle=Swizzle&lt;3,3,3&gt;&gt;;



    let smem_A = SharedMemory::alloc&lt;ATile&gt;();

    let smem_B = SharedMemory::alloc&lt;BTile&gt;();

    let acc: Fragment&lt;MMA_m16n8k16, D, F32&gt; = Fragment::zero();



    // Typed pipeline — double buffer, compiler verifies NoOverlap

    let pipe: Pipeline&lt;stages=2, layout=ATile&gt; = Pipeline::init();



    for k in 0..K step 16 {



        // Async load — produces a linear Transfer obligation

        let tx_A: Transfer&lt;Global, Shared, Async&lt;1&gt;, 128&gt; =

            cp_async(A[k], smem_A);

        let tx_B: Transfer&lt;Global, Shared, Async&lt;1&gt;, 128&gt; =

            cp_async(B[k], smem_B);



        // Discharge obligations — compiler enforces this precedes ldmatrix

        pipe.wait(tx_A, tx_B);   // emits: cp.async.wait_group 1

        barrier::sync();          // emits: bar. sync 0



        // ldmatrix — types verified against smem layout

        let frag_A: Fragment&lt;MMA_m16n8k16, A, F16&gt; = ldmatrix(smem_A);

        let frag_B: Fragment&lt;MMA_m16n8k16, B, F16&gt; = ldmatrix(smem_B);



        // mma — roles verified, acc accumulated in-place

        acc = mma_sync(frag_A, frag_B, acc);

    }



    // Store — compiler knows acc is in registers, C is in global

    store(acc -&gt; C);

}




4.4 Cache Policy Annotations

// Weight tiles — persist in L2 across many activation tiles

@cache_policy(L2_PERSIST, reuse_count=8)

let weights: L2Memory&lt;F16&gt; = load(weight_ptr);



// Activation tiles — one-time use, evict immediately

@cache_policy(L2_EVICT_FIRST)

let activations: L2Memory&lt;F16&gt; = load(act_ptr);



// Emits: ld.global.L2::evict_last  (for weights)

// Emits: ld.global.L2::evict_first (for activations)




4.5 PTX Escape — For When You Need It

// Y-Lang has a PTX escape hatch for edge cases

// Unlike CUDA C's asm volatile, the escape is TYPED

// You declare what registers go in and what comes out



@ptx_emit(

    inputs:  [smem_ptr: SharedMemory&lt;F16&gt;],

    outputs: [r0: reg32, r1: reg32, r2: reg32, r3: reg32]

)

&quot;ldmatrix.sync.aligned.m8n8.x4.shared.b16 {r0,r1,r2,r3}, [smem_ptr];&quot;



// The type system still tracks r0..r3 as register values

// They can be fed into mma_sync with full type checking





5. The AI-Powered Compiler

Y-Lang's compiler has two distinct layers: a deterministic type-checking and code-generation layer that guarantees correctness and an AI layer that optimizes performance. These layers never overlap. The AI suggests, the type system verifies, and code is emitted only if verification passes.





Hard Rule

AI components are advisory, not authoritative. A bad AI suggestion caught by the type checker falls back to a safe default. An AI suggestion that bypasses verification and produces incorrect output would be a language correctness bug — and the architecture makes that impossible by construction.




5.1 Where AI Is NOT Used

The following compiler stages are fully deterministic and involve no AI:





Bank conflict analysis — the type system produces a mathematical proof, not a heuristic



Fragment role verification — phantom types, zero-cost, compile-time only



Transfer obligation tracking — linear type system, formal verification



PTX emission — correct by construction from the typed IR



Register allocation on fixed ISA patterns — solved problem, deterministic optimum



Using AI in these stages would replace proofs with probabilities — a strict downgrade.




5.2 Where AI Is Used




Stage 1 — Tile Size Prediction (Before Compilation)

The hardest unsolved problem in kernel compilation: given a matrix shape, a hardware profile, and an operation type, what are the optimal BLOCK_M, BLOCK_N, BLOCK_K, num_warps, and num_stages?



Triton brute-forces this with an autotuner that can run for hours. Y-Lang instead uses a small transformer trained on (hardware config, kernel shape) → (optimal tile config, measured TFLOPS) pairs, predicting near-optimal configurations in milliseconds.



// AI Tile Predictor — runs before compilation

// Input:  { hardware: RTX_4070, op: GEMM, M: 4096, N: 4096, K: 4096 }

// Output: { BLOCK_M: 128, BLOCK_N: 128, BLOCK_K: 32,

//           num_warps: 4, num_stages: 3, confidence: 0.94 }

//

// If confidence &lt; threshold, falls back to exhaustive search

// Predicted config is still validated by the type system before use




Stage 2 — SASS-Level Hazard Detection (After PTX Emission)

After the deterministic PTX emitter runs, an AI model trained on (PTX, SASS disassembly, performance counter) triples scans the output for patterns that produce poor SASS:



Load-use hazard chains — a load result consumed by the immediately following instruction

Register spill patterns — sequences that will cause the register allocator to spill to local memory

Uncoalesced access signatures — memory access patterns that will serialize across the warp



This is particularly valuable because Y-Lang's target audience already reads SASS. The AI model augments rather than replaces that skill — catching hazards in code paths the programmer didn't manually inspect.




Stage 3 — Autotuning Loop Closure (After Profiling)

The current optimization loop in practice: profile kernel → read Nsight counters → identify bottleneck → infer code transformation → reimplement → reprofile. This cycle requires a human expert and takes hours.



// AI Autotuner reads performance counters and suggests Y-Lang changes

//

// Input:  { sm_active_cycles: 0.61, l2_hit_rate: 0.43,

//           tensor_core_utilization: 0.78 }

//

// Output: { suggestion: 'L2 hit rate below threshold for weight tiles.',

//           action: 'Add @cache_policy(L2_PERSIST) to weight load on line 24.',

//           predicted_improvement: '+12% L2 hit rate',

//           confidence: 0.87 }

//

// Suggestion is applied as a Y-Lang annotation — still type-checked

5.4 Training Data Sources

The AI components do not require novel data collection. Existing public sources provide sufficient training signal for all three stages:





CUTLASS benchmark suite — thousands of pre-tuned kernel configs with measured TFLOPS, directly usable as tile predictor training data



Triton autotuner logs — every benchmark run produces (config, TFLOPS) pairs; publicly available in the Triton repository



NVIDIA MLPerf submissions — optimized kernel configs for specific hardware with measured results



YSU Engine kernels — real SASS analyzed by hand, ground truth for the hazard detector





6. Why Y-Lang May Matter in the Future




6.1 The Trend Toward Specialization

GPU hardware is growing more heterogeneous, not less. The H100 introduced the Transformer Engine with FP8 tensor cores. Blackwell introduced FP4. Each generation adds new instruction types, new memory hierarchy levels, and new warp-collective operations with fixed register layouts. The &quot;write once, run anywhere&quot; promise of portable kernel languages becomes harder to keep as hardware continues to diverge.



Y-Lang bets on the opposite trend: that the highest-value workloads will always be run by people who care enough to target specific hardware — the render farm running inference on 1,000 RTX 4090s, the HPC cluster where every microsecond of latency has a cost, the research lab squeezing one more percentage point of GPU utilization out of a plasma simulation.




6.2 The Type System as a Research Contribution

Static bank conflict analysis is, to current knowledge, not implemented in any production compiler. Triton verifies tile shapes; CUTLASS uses template instantiation errors for shape mismatches. Neither proves conflict-freedom. A compiler that can emit a proof certificate — &quot;this kernel has zero bank conflicts for all valid inputs&quot; — is a publishable result, not merely a language feature.



The linear type system for Transfer obligations is similarly novel in the GPU kernel space. Linear types for memory safety are well-studied in languages like Rust, but applying them to asynchronous GPU memory movement — where the obligation spans a hardware pipeline rather than a thread boundary — is new territory.




6.3 The AI Compiler as a Research Direction

Learned tile size prediction is an active research area. DeepMind's AlphaTensor demonstrated that learned approaches can discover matrix multiplication algorithms that outperform human-designed ones. The Y-Lang tile predictor is a narrower, more tractable version of the same idea: learn the mapping from (kernel shape, hardware spec) to (tile config) from existing benchmark data, then use it to replace hours of autotuning with milliseconds of inference.



The SASS hazard detector is a form of learned static analysis — a field attracting significant recent interest in the program analysis community. Applying it to GPU assembly, where hazard patterns are well-defined, and ground truth is measurable via performance counters, is a concrete and publishable instantiation.




6.4 Potential Use Cases





Render farms and studios: Y-Lang kernels for ray tracing, denoising, and compositing — workloads where the same kernel runs millions of times and every percentage point of GPU utilization has a direct cost impact.



Scientific simulation: Plasma dynamics, fluid simulation, molecular dynamics — workloads where the compute pattern is known in advance and the hardware can be targeted precisely. The YSU Engine's LBM fluid solver and plasma kernels are early examples.



AI inference at scale: Inference serving with bounded latency and a homogeneous hardware fleet. A fleet of RTX 4070s running the same model benefits from a Y-Lang kernel tuned precisely for that hardware, rather than a general CUDA kernel optimized for no hardware in particular.



Compiler research: The type system is a research artifact independent of the language's adoption. The static bank conflict proof, the linear Transfer type, and the fragment phantom type system are each individually publishable contributions.




6.5 Honest Limitations

Y-Lang is not for everyone, and it is not trying to be. The following are genuine limitations, not false modesty:





Architecture-specific: A Y-Lang kernel for the RTX 4070 does not run on an RTX 3060 or H100 without modification. This is a deliberate tradeoff, not an oversight.



High skill floor: The language assumes the programmer understands what ldmatrix is, what a warp-collective operation is, and why fragment roles matter. It is not a teaching tool.



Static analysis boundary: Bank conflict proofs apply to warp-collective operations with fixed access patterns. Arbitrarily indexed SMEM access with runtime strides falls outside the provable domain and requires a runtime fallback or explicit programmer annotation.



AI component limitations: The tile predictor is advisory. For kernel shapes well outside its training distribution, it falls back to exhaustive search. The SASS analyzer may produce false positives on unusual but correct code patterns.





7. Implementation Roadmap

Y-Lang is currently in the design phase. The build order below is chosen to minimize wasted work: each phase produces something runnable before the next phase begins.




Phase 1 — Type System Prototype



Define the five type tiers in a host language (Rust recommended for type system expressiveness)



Implement SmemLayout with Swizzle type parameter



Implement the static bank conflict checker for LDMatrix access patterns



Implement Fragment phantom types with role enforcement



Implement Transfer as a linear type with obligation tracking



Target: a type checker that accepts correct kernels and rejects wrong ones, even before PTX emission




Phase 2 — PTX Emitter



Define the typed IR that the type checker produces



Write a PTX emitter that lowers typed IR to valid PTX



Target: a Y-Lang kernel that compiles to runnable PTX and produces correct numerical output



Benchmark against an equivalent hand-written CUDA C kernel as a baseline




Phase 3 — Benchmark Harness



Build a framework for measuring kernel performance: TFLOPS, L2 hit rate, SMEM throughput.



Collect (kernel config, measured performance) pairs for RTX 4070



This data becomes the training set for Phase 4




Phase 4 — AI Tile Predictor



Augment Phase 3 data with CUTLASS and Triton benchmark results



Train a small model (gradient boosted tree or small transformer) on (shape, hardware) → (tile config)



Integrate predictor output as a pre-compilation pass



Measure prediction accuracy vs. exhaustive autotuning




Phase 5 — SASS Analyzer and Autotuner



Collect (PTX, SASS, performance counters) triples from Phase 3 kernels



Train hazard detector on this data



Build the autotuning loop: profile → AI suggestion → annotate → recompile → verify improvement





Implementation Language

Rust is the recommended implementation language for the compiler. Its type system natively supports linear types (via ownership), phantom types (via PhantomData), and the compile-time symbolic evaluation required for the bank conflict proof. The PTX emitter can begin as a straightforward string emitter in Phase 2 and be upgraded to an LLVM/MLIR backend in later phases.
